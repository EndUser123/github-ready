Workflow Reference

Execution model for /github-ready: package-type detection, scaffolding, validation, media generation, polish, cleanup, and git-ready output. This page is written as an implementation-facing reference, not a marketing overview.

phase pipeline inputs / outputs technical reference

Pipeline graph

The workflow is ordered to front-load deterministic structure generation and quality gates. Media and public-facing polish happen only after the repository is in a valid state.

Phase table

Phase Purpose Primary inputs Primary outputs
1 Normalize local state and inspect the target tree. repository root, existing files, Claude state clean baseline, tree snapshot, package hints
1.5 Determine whether the target is a plugin, skill, library, or brownfield conversion candidate. .claude-plugin/, SKILL.md, src/, pyproject.toml package type decision
2 Create canonical directory structure and essential metadata. package type decision scaffolded directories, config, tests layout
3 Write README, license, documentation, and package-specific templates. scaffold + package metadata public-facing repo documents
4 / 4.5 Run structural checks, then review for quality and cross-file issues. generated repo + codebase validation status, findings, fixes required before polish
4.7 Generate media only after the repository is coherent enough to describe. review bundle, selected source files, NotebookLM inputs image, video, HTML pages, flowcharts, slides
5 / 6 / 7 Apply presentation polish, remove stale artifacts, and prepare the git history. validated repo + generated media publishable GitHub repository

Operational details

Package detection

  • .claude-plugin/ implies plugin mode
  • SKILL.md implies skill mode
  • src/ or pyproject.toml implies library mode
  • library + user confirmation can trigger brownfield conversion

Quality gate

  • validation runs before portfolio polish
  • media is generated after code review, not before
  • critical findings should block polish
  • advisory findings can be documented and carried forward

Media strategy

  • review bundle provides architecture context
  • source files provide implementation detail
  • video brief constrains NotebookLM tone and length
  • GitHub Pages hosts browser-playable artifacts
The media path is intentionally constrained: the generated video should be a short technical walkthrough, not a long presentation or sales-style narration.

Representative commands

/package mylib
/package --dry-run
/package mylib --skip media

# media generation primitives
nlm infographic create "$NOTEBOOK_ID" --orientation landscape --detail standard --style professional --confirm
nlm video create "$NOTEBOOK_ID" --format explainer --style documentary --confirm
nlm slides create "$NOTEBOOK_ID" --slide-format detailed_deck --confirm

# duration verification
ffprobe -v error -show_entries format=duration -of default=noprint_wrappers=1:nokey=1 assets/videos/{package}_explainer_pbs.mp4

Design constraints

GitHub compatibility

  • README diagrams should use simple Mermaid flowcharts
  • avoid C4 blocks in the main README surface
  • do not rely on inline README video playback

NotebookLM inputs

  • exclude generated outputs and caches
  • include architecture summary + implementation files
  • upload a narration brief before video generation

Publishing path

  • README links to GitHub Pages for browser playback
  • release assets remain the direct-download fallback
  • runtime labels should be verified from actual files
This page is intentionally plain. The goal is to answer “how does the pipeline work?” quickly for engineers reviewing the package, not to serve as a landing page.
